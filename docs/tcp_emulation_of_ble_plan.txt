I'd like to make sure this is thoroughly documented and the roadmap is clear, especially so the next few steps are executed with awareness of the broader context. just for thoroughness' sake, I'll at least mention as many of the points I think are important as I can think of: TCP can "back" bluetooth connections as needed; right now TCP is only using configured peers recorded in files at static IPs because we're on tailscale and it works, but there's a multi-tiered plan to implement MDNS (hope I remembered that name right) lookup, or hole-punching if needed, or ultimately a server at rimm.ing (a domain we own) to forward traffic, a list loosely modeled after how Signal does robust networking for peer-to-peer encrypted communication. BLE templates all data transfers between devices and its language is used just below the app level (apps don't need to know how the sync engine works at all, they only need to know about flows, except to the extent that they need to be involved in setup e.g. on mobile but that's ideally one-time and apps-using-soradyne's code isn't littered with references to BLE) throughout soradyne's networking impelmentation, but this template can actually be implemented a variety of ways, one of which is BLE itself for sure, but others of which include TCP and, eventually, CAN bus, LoRa, other protocols. at the device level, devices open and utilize connections to other devices that are phrased as light abstractions over BLE, but don't participate in deciding what kind of connection is actually set up--that is modular, they only read/write/handle exceptions as if it's always BLE, while behind the scenes a heterogeneous mix of connection types may be set up and torn down e.g. a literal BLE connection might get replaced with a TCP if it's discovered that the device currently being addressed over BLE is a peer set up for TCP; at best, this switch-over would have literally no effect whatsoever on the application, the data it's exchanging would just get there one way or another; at worst, soradyne might close the connection, making the app re-connect, which would then be a new connection over TCP instead, but yeah that would not be the best case and the close-connection-rely-on-retry behavior should really just be a fail-over in case fancy multiplexing gets crossed up. that's the near future once we get back to BLE, but for now, between all the containerized tests etc., TCP is backing BLE, just because it's so much easier to develop with. (BLE will ultimately be the main desired channel for all low-bandwidth connections for which it can be used because it's local to the body and therefore more secure under the security models that are relevant to rim/the rim protocol.) event-driven is pretty much always to be preferred over polling, e.g. sync should be triggered by processes taking actions that require data to move, and connections should be maintained by reacting to drops with reconnect attempts in an event-based way, which might involve an exponential back-off or something but not a process that hangs out in a loop as its main modus, if that makes sense. soradyne (according to recent design decisions) runs within apps, via ffi in the case of dart, contrasting with a different model which would be some sort of daemon process or background app that must always be running for rim-protocol sync to run; since TCP can be clingy about its ports, whereas bluetooth is managed and multiplexed by the OS and therefore generally doesn't experience conflicts with different apps sending and receiving information to the same external device, meaning that, in order to surface ONLY BLE functionality to the apps and isolating them from the fact that TCP might (or might not) be implementing the connection, soradyne might need to open multiple TCP connections between the same pair of devices, one+ for each app, to fully replace the functionality of BLE without e.g. forcing apps to deal with TCP-specific exceptions (that kind of error should be handled within soradyne, beneath the level of the simulated BLE connection, and if they aren't recoverable, surfaced as the appropriate corresponding BLE exceptions like just a basic connection failed, that sort of thing). in the case where there is no app on a computer/device but we still want data synced to it, there's a soradyne_cli capability to run a daemon, which basically acts as a surrogate app, but it's optional, and if there were an app running on the device (e.g. a desktop version of giantt), there would be no need for the daemon (this information is provided as a constraint on the design, e.g. you wouldn't want to implement giantt on desktop in a way that requires a running daemon process because that sets up an inconsistent design pattern, making it hard to know based on the examples being built now what the right way to do it in the future with other apps is; instead, you would call the soradyne rust code from within the desktop app, again via ffi for dart, duplicated per-process (making it important to prioritize lightness at least for those parts of soradyne that must live in each app's process memory, possibly by making it load only certain parts e.g. if it isn't going to be syncing images it shouldn't be loading ffmpeg). you might run the soradyne_cli-packaged daemon and an app that syncs with soradyne (e.g. giantt) on the same computer at the same time, if you wanted to monitor the state of flows in terminal, say, but again, just for the sake of reinforcement, if those processes were using bluetooth to sync to the other device(s), they would each have separate bluetooth connections managed/multiplexed by the OS, which is the normal way BLE works, OR, if some of those connections are backed by TCP under the hood, a separate TCP connection would need to be allocated for each process so they aren't throwing TCP-specific errors that would never happen under pure bluetooth (connection refused, port busy etc); soradyne has to manage the multiplexing part that the OS would be doing if it were just bluetooth. I recognize that this introduces some challenging problems around singularity and plurality for listening processes vs connecting processes, possibly necessitating a side-channel whenever TCP is in use that is properly singular per-device-pair just for negotiating how the others should work; design will have to happen there and it may be hard to control complexity, and I'll be seeking advice on this from you (claude) and my network tech savvy friends as to how to get a TCP-backed connection to precisely present as BLE at the app layer despite TCP's limitations.
